COMPREHENSIVE MIGRATION & DEPLOYMENT PLAN: Replit to VPS

Guiding Philosophy: We are not moving a Replit app. We are rebuilding a production-grade Node.js application on a VPS, using the code from your repository as the source. The Replit preprod is a reference implementation, not the blueprint for production.
Phase 0: Preparation & Foundation (The "Stop the Bleeding" Phase)

Goal: Stabilize the current state, gather intelligence, and prepare the artifacts for a clean build.

    Immediate Action on VPS:

        Isolate the Broken Prod: Stop the current systemd service (sudo systemctl stop your-app-service). This prevents further conflicting processes and allows you to work on a clean slate.

        Document the Current State: Take a full backup of the current /etc/systemd/system/your-app.service and your nginx configuration. Then move them to a backup location (/home/ubuntu/backups/). This eliminates their influence while we work.

    Code & Configuration Audit (Local/Preprod):

        Identify Implicit Replit Dependencies: Run npm list --depth=0 on your working Replit preprod environment. Document all direct dependencies. Compare this to your package.json. Look for anything installed globally or via the Replit packager that isn't explicitly listed.

        Pin Your Versions: In your package.json, ensure all dependencies are pinned to exact versions (e.g., "4.18.2" instead of ^4.18.2). This guarantees reproducible builds.

        Create a Single Source of Truth:

            config/production.js: Create a Node.js config file that explicitly sets production variables (database URL, port, etc.) using process.env.

            .env.production: Create this file but add it to .gitignore. This file is for local testing of production configs and should never be committed. It will not be used on the VPS.

        Fix Module Incompatibility: This is critical. Identify the modules causing "Dynamic Require" errors. The solution is often to:

            Use an ESM-compatible version of the library.

            Or, use a dynamic import() statement instead of require().

            Or, as a last resort, transpile your code to CommonJS (see Phase 1).

Phase 1: The Build Pipeline Reconstruction

Goal: Create a reliable, self-contained build process that outputs a runnable artifact completely independent of Replit's tooling.

    Update package.json Scripts:
    json

    {
      "scripts": {
        "dev": "tsx src/app.ts", // Stays for local development
        "build": "tsc && npm run copy-assets", // Compiles TS to JS, copies views, images, etc.
        "copy-assets": "cp -r src/views dist/ && cp -r src/public dist/", // Example
        "start": "node dist/app.js", // Runs the compiled JS
        "clean": "rm -rf dist/" // Clean build artifact
      }
    }

        Key Change: Move away from tsx for production. The VPS will run node dist/app.js.

    Build the Artifact (in CI or Locally):

        Run npm run clean && npm install --only=prod && npm run build. This creates a ./dist folder containing only compiled JavaScript and necessary assets.

        Test this artifact locally: Run npm run start from the project root. Does it start without errors? This proves the code and build process are sound.

Phase 2: VPS Environment Provisioning (The "Bare Metal" Setup)

Goal: Create a clean, secure, and consistent runtime environment on the VPS. Do this on a fresh VPS if possible, or meticulously clean the current one.

    Create a Dedicated System User:
    bash

sudo adduser --system --group --no-create-home appuser
sudo mkdir /opt/your-app
sudo chown appuser:appuser /opt/your-app

    This enhances security by not running your app as root.

Install & Configure PostgreSQL on the VPS:

    Install: sudo apt install postgresql postgresql-contrib

    Create a dedicated user and database:
    bash

    sudo -u postgres psql
    CREATE DATABASE your_app_prod;
    CREATE USER your_app_user WITH ENCRYPTED PASSWORD 'a_strong_password';
    GRANT ALL PRIVILEGES ON DATABASE your_app_prod TO your_app_user;
    \q

    Import your schema and data from your preprod Neon DB using pg_dump and psql.

Install Node.js via Version Manager (NVM):

    This ensures you control the Node version, independent of the OS package manager.

bash

    curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
    source ~/.bashrc
    nvm install 20 # Use your required LTS version
    nvm alias default 20

Phase 3: Deployment & Orchestration

Goal: Automatically get the built artifact onto the VPS and run it as a robust, managed service.

    CI/CD Pipeline Enhancement (e.g., GitHub Actions):

        Add a job that runs on a push to main (or a prod release tag).

        Job steps:

            Checkout

            Setup Node.js

            Install Dependencies: npm ci (for a deterministic install)

            Run Tests

            Build: npm run build

            Deploy Artifact: Use rsync or scp to copy the entire dist/ folder and the node_modules/ folder to /opt/your-app/ on the VPS.

            Reload Service: SSH into the VPS and trigger a service reload.

    Craft the Robust systemd Service File (/etc/systemd/system/your-app.service):
    ini

[Unit]
Description=Your App Name
After=network.target postgresql.service

[Service]
Type=simple
User=appuser
Group=appuser
WorkingDirectory=/opt/your-app
# The most important line: Environment variables are set HERE, not loaded from a file.
Environment=NODE_ENV=production
Environment=PORT=3000
Environment=DATABASE_URL="postgresql://your_app_user:password@localhost:5432/your_app_prod"
ExecStart=/home/ubuntu/.nvm/versions/node/v20.X.X/bin/node dist/app.js
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target

    Critical Note: All secrets and configs are injected directly into the service environment. This is the VPS standard. The .env file is eliminated.

Configure nginx as a Reverse Proxy:

    Create a config file: /etc/nginx/sites-available/your-app

nginx

    server {
        listen 80;
        server_name yourdomain.com www.yourdomain.com;

        location / {
            proxy_pass http://localhost:3000; # Matches the PORT in systemd
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection 'upgrade';
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_cache_bypass $http_upgrade;
        }
    }

        Enable it: sudo ln -s /etc/nginx/sites-available/your-app /etc/nginx/sites-enabled/

        Test config: sudo nginx -t

        Reload nginx: sudo systemctl reload nginx

Phase 4: The Final Migration Execution

Goal: Perform a controlled, monitored cut-over.

    Final Verification:

        On the VPS, navigate to /opt/your-app.

        Manually run NODE_ENV=production PORT=3000 /home/ubuntu/.nvm/versions/node/v20.X.X/bin/node dist/app.js.

        Does it start? Can you curl http://localhost:3000 from the VPS and get a valid response? This is the final smoke test.

    Enable and Start the Service:
    bash

sudo systemctl daemon-reload
sudo systemctl enable your-app.service
sudo systemctl start your-app.service

Monitor Relentlessly:
bash

    sudo journalctl -u your-app.service -f --lines=100  # Follow the logs in real-time
    curl http://localhost:3000/api/healthcheck          # Test from the server itself

        Watch for any errors. The service is configured to restart on crash, but the logs will tell you why it crashed.
