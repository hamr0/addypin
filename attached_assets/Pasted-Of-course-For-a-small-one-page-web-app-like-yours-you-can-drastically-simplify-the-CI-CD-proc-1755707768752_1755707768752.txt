### The Ultra-Simple CI/CD Pipeline for AddyPin

**Core Principle:** `main` is king. It is always production-ready. Every change is made through a Pull Request and goes live with one click after tests pass.

**Your Simplified Architecture:**

```mermaid
flowchart TD
    A[Make a change on a<br>short-lived feature branch] --> B[Open a Pull Request to `main`];
    B --> C[GitHub Actions AUTOMATICALLY runs:<br>1. Lint<br>2. Unit Tests<br>3. Build Check];
    C --> D{Merge to main};
    D --> E[GitHub Actions AUTOMATICALLY<br>deploys to production];
    E --> F[Run a quick health check];
    F --> G{Is the site alive?};
    G -- Yes --> H[✅ Done!];
```

### Step-by-Step Implementation Plan

#### 1. Repository Structure (Simple)

*   `main` (Protected): Your only important branch. This is what's running in production.
*   `feature/quick-fix`: Short-lived branches for every change. Delete them after merging.

#### 2. The One & Only GitHub Actions Workflow

Create a file in your repo at `.github/workflows/deploy.yml`. This one file does everything.

```yaml
name: Deploy to Production

on:
  # Run tests on every PR to main
  pull_request:
    branches: [main]
  # Deploy automatically when code is merged to main
  push:
    branches: [main]

jobs:
  test:
    # This job only runs on Pull Requests
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Run linter
        run: npm run lint # Or `npx eslint .` if you don't have a script

      - name: Run tests
        run: npm test

      - name: Build the app
        run: npm run build # This ensures the build doesn't fail

  deploy:
    # This job only runs on a push to main (after a PR is merged)
    if: github.event_name == 'push'
    runs-on: ubuntu-latest
    # Add a manual approval gate if you want. For speed, I recommend automatic.
    # environment: production
    # needs: test

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install dependencies
        run: npm ci

      - name: Build the app
        run: npm run build

      # THE KEY STEP: Deploy via SSH
      - name: Deploy to VPS
        uses: appleboy/ssh-action@v1.0.0
        with:
          host: ${{ secrets.SSH_HOST }} # e.g., your.vps.ip
          username: ${{ secrets.SSH_USERNAME }} # e.g., deploy
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            # Navigate to your app directory
            cd /opt/addy-pin

            # Pull the latest code from the main branch
            git fetch origin
            git reset --hard origin/main

            # Install dependencies for production
            npm ci --omit=dev

            # **CRITICAL: Restart the application**
            # If using PM2 (recommended):
            pm2 restart addy-pin --update-env

            # If using systemd:
            # sudo systemctl restart addy-pin

      # Optional but highly recommended health check
      - name: Verify Deployment
        run: |
          curl -f https://yourdomain.com/api/health || curl -f https://yourdomain.com/ > /dev/null || exit 1
```

#### 3. VPS Setup (The Simple Way)

1.  **Create a Deploy User:** On your VPS, create a user (e.g., `deploy`) with limited permissions. Add your public SSH key to its `~/.ssh/authorized_keys` file.
2.  **Clone Your Repo:** As the `deploy` user, clone your repository into `/opt/addy-pin`.
3.  **Setup the App:**
    *   Run `npm ci --omit=dev` to install production dependencies.
    *   **Use PM2:** It's the simplest way to manage Node.js processes.
    ```bash
    npm install -g pm2
    pm2 start npm --name "addy-pin" -- start
    pm2 save
    pm2 startup # This makes it restart on reboot
    ```
4.  **Secrets:** Store your production environment variables (like database URL, API keys) in the `deploy` user's `~/.bashrc` or a `.env` file in `/opt/addy-pin`. Your SSH script will have access to them.

#### 4. GitHub Secrets

Go to your GitHub repo **Settings > Secrets and variables > Actions**. Create these secrets:
*   `SSH_HOST`: Your VPS IP address.
*   `SSH_USERNAME`: The username (e.g., `deploy`).
*   `SSH_PRIVATE_KEY`: The **private** SSH key of the key-pair you added to the VPS.

### Your Safety Net (Simplified Rollback)

With this simple setup, your rollback strategy is beautifully straightforward:

1.  **Git is your rollback tool.** If a deployment breaks, simply revert the merge commit on the `main` branch and push.
2.  The `push` event will trigger the deployment workflow again, pulling the previous, known-good code and restarting the app.
3.  You're back online in under a minute.

**For the database:** Since your app is mostly about reading shortcodes and displaying maps, your critical data is the URL mappings. **Ensure you have a daily automated backup of your database** (e.g., a `pg_dump` cron job that uploads to Backblaze B2 or AWS S3). This is non-negotiable but separate from CI/CD.

### Summary: What's Best for Your Case

*   **Forget `develop` and `staging` branches.** They are overhead you don't need. Use `main` and feature branches.
*   **Use One GitHub Actions File** that both tests PRs and auto-deploys to production.
*   **Your "Staging" environment is your local machine.** Test thoroughly there before pushing.
*   **Rollback via Git Revert.** It's fast and effective.
*   **Automate your database backups separately.**

This system gives you the safety of testing and automated deployments with the speed and simplicity a solo developer or small team needs. You can always add a manual approval step in the GitHub Actions `deploy` job later if you want, but for now, automatic is the way to go.
